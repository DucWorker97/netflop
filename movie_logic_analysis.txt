LOGIC ANALYSIS: MOVIE CRUD OPERATIONS (NETFLOP)
================================================

1. OVERVIEW
-----------
The movie management system in Netflop Admin follows a "Metadata First, Media Second" approach. 
- Creation is a lightweight process that initializes a database record.
- Media upload (Video, Poster, Subtitles) happens asynchronously after the movie record exists.
- The interface has been unified into a single "Edit Movie" page with tabs: [Metadata] [Media & Assets] [Settings].

2. ADD MOVIE (CREATE)
---------------------
File: apps/admin/src/app/(dashboard)/movies/new/page.tsx
Backend: POST /api/movies

[Logic Flow]
1. User enters Basic Info: Title (required), Description, Release Year, Duration.
2. User selects initial Genres.
3. User adds Actors (Draft UI only - currently mapped to no-op or future request, need to confirm backend support).
4. User selects Poster (Client-side preview only).
   - NOTE: The current implementation sends metadata FIRST. The poster selected in step 4 is NOT uploaded during the initial POST. It is expected to be uploaded in the next step (Media Tab), but the UX in `new/page.tsx` implies it might handle it.
   - *Current Behavior*: The `onSubmit` function calls `createMutation` with metadata only. The selected poster file is currently IGNORED in the initial create call. The user must re-select the poster in the Media tab.
   - *Post-Creation*: Redirects to the Edit page (Bug identified: redirects to legacy `/media` route).

[Status Transitions]
- Initial `movieStatus`: 'draft'
- Initial `encodeStatus`: 'pending'

3. EDIT MOVIE (UPDATE)
----------------------
File: apps/admin/src/app/(dashboard)/movies/[id]/page.tsx (Container)
Tabs: MetadataTab, MediaTab, SettingsTab

A. METADATA UPDATE
------------------
Backend: PUT /api/movies/:id
- Updates simple fields (title, etc.) directly.
- Updates Genres: Performs a "Delete all existing relations -> Insert new relations" transaction.
- Triggers: `aiService.triggerRetrain()` to update recommendation embeddings.

B. MEDIA OPERATIONS (VIDEO)
---------------------------
Backend: PUT /api/upload/presigned-url -> S3 -> POST /api/movies/:id/upload-complete
1. Presigned URL: Client requests secure upload URL for `video/mp4` etc.
   - Generates unique key: `originals/{movieId}/{uuid}-{filename}`.
2. Upload: Client uploads directly to S3/MinIO via PUT (bypass API server).
3. Completion: Client calls `uploadComplete`.
   - Server verifies S3 object exists (implicitly or trusting client).
   - Server creates `EncodeJob` record.
   - Server enqueues `ENCODE_HLS` job to BullMQ.
   - Update `encodeStatus` -> 'pending'.

C. MEDIA OPERATIONS (POSTER)
----------------------------
Backend: Same Presigned flow -> `posters/{movieId}/...`
- Completion: Updates `posterUrl` column in `Movie` table directly to the public URL.
- No encoding step for posters.

D. SUBTITLES
------------
Backend: PUT /api/upload/subtitle-presigned-url
- Uploads to `subtitles/{movieId}/...`
- Completion: Updates `subtitleUrl` column.
- Current Limitation: Supports single subtitle file in `subtitleUrl` column or array in `subtitles` (implementation varies, code shows `subtitles` array in interface but schema might be simpler).

4. DELETE MOVIE
---------------
File: apps/admin/src/app/(dashboard)/movies/[id]/_components/SettingsTab.tsx
Backend: DELETE /api/movies/:id

[Logic Flow]
1. User clicks "Delete Movie" -> "Are you sure?" confirmation.
2. API executes `prisma.movie.delete({ where: { id } })`.
3. Database constraints (Cascade) delete related `MovieGenre`, `WatchHistory`, `Reviews`.

[Critical Gap]
- **S3 Cleanup**: The current `MoviesService.delete` method does NOT delete the associated S3 objects (`originals/...`, `hls/...`, `posters/...`).
- **Consequence**: "Orphaned" files remain in storage, consuming space indefinitely unless an external garbage collector runs.

5. ANALYSIS CASES
-----------------

[Case 1: Happy Path Creation]
- User creates "Inception".
- DB: Created. Status: Draft.
- User redirects to Edit -> Media.
- User uploads Video. Worker encodes. Status: Processing -> Ready.
- User uploads Poster.
- User publishes. Status: Published.

[Case 2: Re-uploading Video]
- Admin wants to fix a glitchy video.
- Clicks "Replace Video" in Media Tab.
- New file uploaded -> New unique key generated.
- `uploadComplete` triggers new Encode Job.
- `encodeStatus` resets to 'pending' -> 'processing'.
- Old HLS files are overwritten by output prefix logic `hls/{movieId}/` (Worker logic usually clears directory).
- *Gap*: The *old* original video file in `originals/` remains (orphaned).

[Case 3: Validation Errors]
- Uploading >500MB file (if configured limit).
- `UploadService` checks size limits before granting presigned URL.
